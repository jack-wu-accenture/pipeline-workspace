# EBX替换与MDM规划评估_summary


---

## 🧭会议概览
本次讨论围绕“是否继续使用 EBX，以及是否以最小代价替换 EBX、构建新的 MDM 方案”展开，核心是在多个可选路线之间形成可执行的规划框架。与会者重点比较了两条主线：一是保持现状，继续使用现有 EBX 体系；二是在尽量少改动外围能力的前提下，把 EBX 的核心能力替换为新的 MDM 底座，可能是自建、AI 辅助构建，或借助其他平台承载。讨论中反复强调，当前不应过早陷入“用什么技术、是否用 AI、是否采用开源产品”的枝节，而应先建立评估标准、确认 Global 接受的准入条件、梳理现有架构中哪些能力必须保留，再基于成本、周期、风险、预算和业务影响做方案判断。整体上，这是一次偏前期规划和决策准备的会议，目标不是直接拍板技术实现，而是明确后续分析框架、输入材料和决策路径。

## 🔑会议关键点
### 🧱方案范围与决策框架
- **核心决策问题**：当前真正需要比较的方案并不多，本质上只有两条路线。第一条是继续沿用现有 EBX，保持现状；第二条是尽量抽掉 EBX 的“壳”或核心依赖，转向新的 MDM 承载方式。会议中明确提出“其实就是两个方案，没有第三个”，说明当前讨论不应无限发散。
- **评估维度**：所有方案都必须综合考虑稳定性、时间成本、人力成本、License 成本、实施周期、业务影响、预算、风险以及是否能满足既定 Deadline，而不是只从技术实现难易度出发。
- **方案命名澄清**：讨论中将“保持现状”理解为 Keep as is；将“从 EBX 迁移到 New MDM / AI 驱动的 MDM，但尽量最小改动，仅做 Replace”的路线视为 Plan B。该 Plan B 的关键不是完全重构，而是以 Minimal change 的方式替换核心部件。
- **当前阶段定位**：本阶段更偏 Planning，不是立即进入开发实施。Planning 需要拆解成若干任务，先把该验证的东西验证清楚，再决定后续实施路线。

### 🏗️新 MDM 方案应具备的能力框架
- **不能直接无框架自研**：与会者认为，新方案不能简单理解为“自己开发一个系统”，而需要先有一套能力框架，否则无法做系统性评估。
- **基础能力分层**：讨论中提到至少要从几个层面梳理新 MDM 的基础能力，包括 Data Storage（数据存储层）、Data Lineage（数据血缘/关系追踪）、Security（安全）等。这意味着未来评估不是只看数据库替换，而是要看完整的平台能力是否覆盖。
- **质量与治理能力**：会议中特别提到 DQ（Data Quality，数据质量）相关问题，认为无论是现有体系还是未来 MDM，都必须具备质量管理能力。要进一步明确，当前哪些质量能力与 EBX 强绑定，未来若迁移到新系统，这些能力如何继承或重建。
- **评估文档需求**：会上明确有人提出“这个东西我写不出来，需要你出一个”，说明需要形成一份结构化评估说明文档，由相对更熟悉框架的人执笔，作为后续沟通和决策输入。

### 📏评估标准与 Global 认可问题
- **评估标准是第一条线**：会议明确指出，项目要走两条线并行推进，其中一条就是评估标准线。必须先找到“什么样的方案才算合格”，否则任何技术方案都缺乏评判基准。
- **关键问题不是团队自定义五个标准**：有与会者指出，当前有人列了五个标准，但 Global 并不认可；如果这些标准真被认可，对方自己就会推动了。说明现在的问题不是“有没有标准”，而是“谁认可这个标准、标准从哪里来”。
- **应获取真实 Criteria**：讨论反复聚焦在“Global/EDM 现在真正要求的 Criteria 是什么”。如果已有标准，需要拿来作为输入，从标准反推未来 New MDM 应该怎么设计，而不是先设计再找理由包装。
- **责任与背书问题**：Bruce 对某些框架有自己的理解，但 Global 不确认，原因之一是担心承担责任。因此需要弄清是否其实已经存在一套框架，只是当前没人明确背书。
- **关键联系人识别**：讨论中提出需要拿到 DKM 相关材料，并识别“负责 Global 的印度确认人是谁”，与其直接沟通评估标准。这说明接下来需要通过关键人访谈获取正式或半正式的准入条件。
- **最终目标**：在拿到标准后，需要明确计划中采用什么方法与对方达成一致，满足对方红线，最终做整体评估与决策，并落到 Roadmap。

### 🛠️现有 EBX 架构拆解与替换边界
- **不是简单的“换或不换”**：会议认为，EBX 并不是单一产品，而是在其外部叠加了大量自开发工作和外挂逻辑，因此不能把它视作一个可整块替换的黑盒。
- **现有架构理解**：从讨论看，EBX 至少涉及底层数据库、服务层、外围 API/外挂逻辑等部分。有人认为 DB 一定需要被替换；服务层承担数据进入、写入和流转的功能；而大量自开发逻辑很可能并不在 EBX 内部，而是在外层外挂中实现。
- **保留外挂的可能性**：讨论中有人判断，某些外围逻辑层可以保留，只需要重新对接到新的底层或服务层上。如果外挂可保留，改造规模就可能从原先估计的 75% 降到 35% 左右。
- **关键难点**：真正需要攻克的，是 DB 与 API/服务之间那一层被 EBX 强绑定的能力，以及现有外挂究竟承担了哪些逻辑。如果这一层被明确拆开，替换难度和影响面都将明显下降。
- **实施影响理解**：如果未来不再保留现有“中间车”，则数据库将与 API 更直接对接，API 需要相应改造。这部分应纳入技术方案设计，而不是只做概念讨论。

### 💰成本、平台与替代路线争论
- **低成本导向明显**：讨论透露出推动变更的核心驱动之一是成本，尤其是 License 成本。会议中提到相关费用从 450K 到 900K 的压力，Global 采购又谈不下来，这构成了替换 EBX 的现实背景。
- **对直接手搓方案的质疑**：部分与会者认为，若不想继续用 EBX，更合理的顺序应是先看成熟产品或已有平台承接，例如 Salesforce 上的 MDM 能力，而不是一上来就直接自建甚至在 AWS 上手搓。
- **平台公信力问题**：讨论中明确区分了 Global 产品、Local 产品与自建方案的可信度。与会者担心 Local 产品在 Global 那里缺乏公信力，而微软、Salesforce 之类常规企业平台相对更容易被接受。
- **已有资产复用**：如果基于 Salesforce 等现有平台承接，一方面可能已经有 License，无需额外支付；另一方面也避免从零手搓 MDM 底座，理论上更有利于节约成本和缩短周期。
- **上游下游改动争议**：有人担心若切到其他平台，上游接口需要改；但也有人反驳称大量上游 DCR 数据本就在现有负责范围内，数据整合与接口改造并没有想象中复杂，甚至已有部分评估材料可复用。

### 🤖AI、自建与风险控制讨论
- **AI 只是选项，不是当前结论**：会议提到 AWS 等外部厂商推荐用 AI 来“搓”方案或产品，但与会者总体态度审慎，认为这只能作为一个可了解的 Option，不能成为当前讨论核心。
- **反对“为了 AI 而 AI”**：会议中明确提出，不要为了做 AI 而做 AI。应先把目标、时间、方式、范围定清楚，再决定 AI 在其中扮演什么角色。
- **AI 代码可信性担忧**：与会者普遍担心 AI 生成的代码即使表面可运行，也不代表能够正式上线。上线需要有人验证、背书、承担责任，而不能把责任转嫁给 AI。
- **可维护性问题**：大家担忧未来如果代码由 AI 大量生成，交接到运维或其他团队时，可能没人真正看得懂。一旦出问题，仍然需要人工回头验证、排查和修复。
- **测试压力风险**：有人判断，如果过度依赖 AI，项目后期很可能变成“大量测试去找问题”，而不是通过高质量设计和开发把风险前移。这会导致实施阶段成本和不确定性进一步上升。
- **Planning 与实施的边界**：会上形成一个较清晰观点——Planning 阶段主要验证方案可行性；至于实施周期中 AI 如何参与开发，那是后续再细化的问题，不应在当前阶段抢占主要讨论空间。

### 🌐外部输入与厂商方案
- **AWS 推荐方案**：会上提到 AWS 在峰会及销售沟通中向团队推荐了一个方案/产品，并被转给相关人员评估。当前对其理解还较模糊，只知道是某种“很强、似乎什么都能做”的产品，甚至被描述为接近万能。
- **对推荐方案保持警惕**：与会者担心厂商推荐带有明显销售导向，例如 AWS 更关心后续服务器和云资源消耗，而非真正替项目承担架构和治理责任。
- **仍值得纳入备选评估**：虽然对开源或厂商推荐方案持保留意见，但大家也意识到，如果完全不提这些选项，未来可能反被质疑“是否评估过其他替代路线”。因此合理做法是纳入了解范围，但不让其扰乱当前主线。
- **当前工作重点不变**：讨论中多次强调，眼下应先解决标书/评估框架问题，不要被尚不成熟的产品宣传带偏。

### 👥组织、职责与内部推动关系
- **项目推动存在角色边界**：有人表示，原则上自己不应主动介入该项目，因为这并非自己的 Cap 范围；若参与，必须是受托推动，否则容易被误解为“抢地盘”。
- **Kevin 新加入后的变量**：会议提到 Kevin 刚加入，后续可能有新的想法；如果他看到项目最终走向“直接手搓”，可能会有较大反应。这意味着组织层面的认知统一也很重要。
- **David Li 的倾向**：根据讨论，David Li 更偏向 Low cost，因此一直推动相关人寻找更低成本的替代路径，也要求提供基于 Salesforce 的解决方案输入。
- **历史评估资产存在**：会议中提到，曾让相关人员做过技术评估，包括评估结果、Technical barrier、解决方案等，说明团队并非完全从零开始，后续可回收利用既有材料。

## ✅已达成的共识
- 当前实质上以两条主方案为主：继续使用 EBX，或以最小代价替换 EBX 的核心能力，构建新的 MDM 承载方案。
- 现阶段重点应放在 Planning 与可行性验证，而不是马上进入实施细节。
- 必须先建立或获取被 Global/EDM 认可的评估标准，再据此反推目标方案要求。
- 新方案评估必须覆盖稳定性、成本、周期、风险、预算、业务影响、Deadline 等综合因素。
- AI 可以作为备选手段或外部方案来源之一，但不是当前决策核心，也不能替代人工验证与责任背书。
- 需要把现有 EBX 架构拆清楚，尤其是数据库、服务层、API 层、外挂逻辑之间的边界，才能判断真正替换范围。

## ⚠️未达成共识 / 仍需进一步确认
- Global/EDM 目前正式认可的评估标准到底是什么，是否已有成体系框架但无人背书。
- DKM 相关材料具体是什么、由谁持有、谁是可直接沟通并能确认标准的关键负责人。
- 新 MDM 的最优承载路径究竟是自建、AI 辅助自建、Salesforce 平台承接，还是其他产品路线。
- AWS 推荐的产品或方案是否具有实际可行性，是否能通过企业内严格审核，尤其在开源合规与治理方面是否可接受。
- 现有外挂逻辑、服务层与 EBX 的依赖关系到底多深，能保留多少，改造量是接近 75% 还是可压缩到 35%。
- 若采用 AI 参与后续实施，验证责任、代码可维护性、运维交接方式如何设计。

## **🤖AI建议**
- **先标准、后技术**：建议优先把 Global/EDM 认可标准拿到手，再讨论技术路线。否则团队会在 EBX、自建、Salesforce、AWS、AI 等多个方向上反复摇摆，难以形成决策闭环。
- **先拆现状、再谈替换**：应把现有 EBX 架构按“底座、服务、接口、外挂、治理能力”拆解成能力清单，明确哪些是必须保留、哪些是可替换、哪些是历史包袱。只有这样，所谓“最小代价替换”才有落地基础。
- **把 AI 放到正确层级**：建议把 AI 定位为加速分析、原型验证、代码辅助或测试辅助工具，而不是当前项目立项和架构选择的前提。这样既能利用 AI，又能避免责任边界失控。
- **形成一页式决策对比表**：建议后续输出管理层可读版本，将每个方案在成本、周期、风险、可被 Global 接受度、对上游下游影响、对内部资源要求等维度进行打分对比，便于尽快形成方向性决策。
- **重视组织背书与沟通顺序**：该项目不仅是技术问题，也是组织协同与责任归属问题。建议明确谁负责出框架、谁负责对 Global、谁负责技术验证、谁负责后续业务沟通，避免方案合理但推进失效。
