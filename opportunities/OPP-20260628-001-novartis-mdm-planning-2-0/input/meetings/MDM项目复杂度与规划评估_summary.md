# MDM项目复杂度与规划评估_summary


---

## 🧭会议概览
本次会议围绕诺华 MDM / EBX 相关主数据管理方案展开，重点讨论了当前 DCR 流程、HCP/HCO 数据处理机制、批量处理与批量审核、匹配与合并逻辑、第三方审核模式、系统复杂度来源，以及后续投标/规划阶段应如何向客户阐明项目复杂度与可行路径。会议一方面由熟悉历史项目的人回顾当年的系统设计和定制逻辑，另一方面从售前与方案规划角度评估：如果客户坚持重做，应该如何做 Planning；如果复杂度与成本过高，是否应考虑保留 EBX、仅优化方案与成本结构。整体基调是：当前项目不能只凭回忆做判断，必须尽快获取历史文档，最好进一步看到代码，才能形成更扎实的复杂度判断、工期估算与方案建议。

## 🔑会议关键点
### 🏗️ 模块一：当前主数据/DCR整体架构与处理范围
- **下游系统对 Mapping 的依赖**：下游系统会利用 Mapping 识别 MDM 发布的数据，并转换为各自系统内部的数据模型。这说明 MDM/主数据平台不仅是存储层，也承担了一定的数据发布标准化角色，下游对其字段映射和模型语义有依赖。
- **DCR 的主要对象范围**：当前讨论的 DCR 主要围绕 HCP、HCO 及其相关对象展开，基础操作包括 Create、Update、Merge。除主体对象外，Affiliation 这类关联关系也支持 Create 和 Update，说明模型中不只是维护主实体，还维护实体之间的重要业务关系。
- **标准功能与定制功能的边界**：会议中明确指出，很多场景并非依赖 EBX 标准能力完成，而是通过 Spring Boot / Spring Cloud 等外部定制方式实现。也就是说，产品本身只提供底座，真正支撑业务的复杂逻辑很大程度上在主数据层外围或扩展层中完成。
- **Master Layer 的角色**：批量处理场景中，如果批量 Match 后没有识别到重复数据，就会直接创建或更新 HCO/HCP，意味着到 Master Layer 更多是直接刷数据库，而不是再进入复杂的交互或人工处理流。这反映出系统存在“标准审核流”和“快速批处理流”两种不同处理路径。

### 📦 模块二：Batch DCR 的设计背景与特殊性
- **Batch DCR 的业务背景**：Swift 场景下，每个月在特定窗口期会集中清洗销量数据，因此会在短时间内出现大量 HCO 需要 Create 或 Update 的需求，由此产生了 Batch DCR 场景。其本质是业务高峰期集中导入、集中修正主数据。
- **Batch DCR 的技术实现特殊性**：该场景被认为非常复杂，因此没有完全走标准 DCR 模式，而是放在外部服务中处理，依托 Spring Boot / Spring Cloud 承担批量 Match、批量校验、批量创建/更新。
- **批量场景与普通单笔场景差异**：普通 DCR 更接近逐笔提交、逐笔审核；Batch DCR 则强调大批量、高效率、窗口期处理，不适合完全复用单笔审核交互逻辑，因此历史上做了特殊设计。
- **HCO 与 HCP 在批量场景中的差异**：HCO 很多数据在进入系统前就已在外部校验好，因此在系统内更多是落库和去重处理；HCP 则保留更多系统内审核与人工判断流程。

### 🔁 模块三：版本、历史与逆向/回滚能力
- **无专门“回滚”功能**：会上明确指出，系统没有专门开发版本回滚或逆向处理能力。当前支持的是基于已有 History 能力查看历史属性值，而不是提供完整业务回退机制。
- **History 的使用方式**：历史版本能力是平台既有能力，会记录属性历史值，因此从数据角度看可以回到历史某个节点进行查看或理解变更，但不等价于业务意义上的一键回滚。
- **逆向场景支持有限**：与正向的 Create/Update/Merge 相比，逆向纠错、撤销、投入产出式的回退处理并未做特别功能开发，这意味着系统在治理设计上偏向“正向加工”，而非“闭环纠错”。
- **结论**：如果客户未来关注版本治理、回滚审计、误操作恢复，则现有方案明显能力不足，可能成为规划中的补强点。

### 👥 模块四：Batch Review 与第三方审核工作模式
- **Batch Review 的来源**：第三方服务商在旧系统时代就采用“导出待审核数据到文件、在自己侧辅助处理、再批量回传结果”的方式。因此新系统也保留了这套工作模式。
- **具体审核流程**：第三方会下载当天需审核的 DCR 文件，在外部数据库或工具中做辅助判断，例如使用类似 VLOOKUP 的方式比对，再将结果导入系统。每个 DCR 必须有结论：通过、不通过，或 Pending。
- **系统支持范围**：Batch Review 主要支持 HCP 的 Create、Update、Merge 等场景，说明 HCP 的人工校验与审核流程是系统内复杂度的重要来源。
- **Undetermined 状态的引入**：由于业务 SLA 是 3 天内第三方需完成审核，但现实中可能无法判定，因此增加了 Undetermined（待定）状态。审批结果由 Approved / Rejected 扩展为 Approved / Rejected / Undetermined。
- **Undetermined 的业务含义**：当第三方无法确认时，DCR 会退回给前端代表，提示该记录待定，代表可以补充排班表等证明材料，再次提交以辅助第三方确认。这体现出系统设计考虑了真实审核中的“不确定性”，而不只是二元通过/拒绝。
- **系统使用人群规模**：系统本身并非销售直接使用，而是主要供第三方审核人员与诺华内部管理员使用，人数很少，历史上估计在 10 人以内。第三方可能四五人，内部管理员则更少。

### 🔎 模块五：HCP/HCO 审核边界与 Match 机制
- **HCO 与 HCP 审核边界不同**：HCO 数据大多在系统外已经校验完成，进入系统时主要处理 Create 时发现 Duplicate 的情况；HCP 则更多在系统内进行单笔审核或批量审核。
- **EBX Match 的两步式设计**：Create DCR 时才会进行 Match。第一步先用医生姓名、科室名、医院名等信息到 Elasticsearch 中用较宽松条件找疑似 Duplicate Candidate；第二步再调用 EBX 的 Match 能力对候选记录进行打分。
- **Match 结果的用途**：Match 打分只是给人工审核做参考，不会自动决定接受或拒绝，也不会系统自动选择现有记录直接覆盖。设计原则是“中国区永远要人来处理”，保留人工决策。
- **界面呈现需求复杂**：候选重复数据会在界面中并排展示，需要标识哪些属性相同、哪些差异大，还要显示打分并让审核人选择。这导致不仅后端逻辑复杂，前端交互也复杂。
- **性能与架构因素**：会议中推测这种两步式设计可能与 EBX Match 性能不佳有关，因此在架构上引入 Elasticsearch 做初筛，以减少平台原生 Match 的负担。

### 🏥 模块六：数据模型范围与关联关系维护
- **机构对象类型**：HCO 不只包括医院，也包括药房、经销商等几类机构对象，说明机构主数据的类型较丰富。
- **院店关系等关联关系存在**：医院与药店之间存在业务关联关系，例如“院边店”类关系，这类关系会影响销量，因此属于重要主数据内容，需要在系统中维护。
- **关联关系联动的复杂性**：如果医院或药店信息发生变化，相关联关系也可能需要同步调整。虽然系统没有实现基于距离等复杂校验规则，但关系维护本身已带来联动处理复杂度。
- **为何没有做更多规则校验**：会议中明确说明，很多这类规则在 Swift 等上游系统中已经校验过，因此主数据系统被尽量定位为“承接经过验证的数据并保证基本数据质量”，而不是重复做过多业务校验。
- **客户对主数据系统的定位**：HCO 侧甚至曾有想法把 MDM 仅作为数据库使用，导入时都不想做查重；最终因需向全球解释数据质量控制，才保留了最低限度的去重机制。可见客户并不希望在 MDM 中嵌入过多业务逻辑。

### 🧩 模块七：项目复杂度来源——不在“场景数”，在“子场景+定开+界面+平台限制”
- **复杂度并非只来自流程数量**：与会者指出，表面上主流程可能十几个到二十个，但每个流程下面还有大量子场景、边界情况和状态变化，乘起来就可能变成几十甚至上百个细分处理点。
- **批量下载/上传/审核链路复杂**：例如批量下载 DCR、外部处理、再回传审核结果，本身就打破了标准在线审核模式。
- **Merge 的复杂性极高**：HCO/HCP Merge 不只是两个主记录合并，还涉及科室、医生、Affiliation 等关系联动。举例来说，两个医院合并后，其下属科室和医生关系也要重新归并。
- **平台标准能力不足导致大量定开**：这些逻辑并非 EBX 标准功能直接支持，因此当年都需要团队自行编写代码实现，包括如何挂接科室、如何合并科室、如何处理附属关系等。
- **界面侧复杂度同样高**：除了后端流程，界面也要支持复杂比对、属性高亮、候选选择、Winner/Loser 属性切换等，属于重度定制。
- **复杂度被低估的风险**：会上多次提到，单从口头描述看，客户可能只觉得“有一些场景”，但难以感受到为什么历史项目会花到数百万级别，因此必须通过文档与事实还原复杂度。

### 🥇 模块八：Winner/Loser 合并逻辑与人工干预
- **Winner/Loser 不是绝对保留/删除关系**：Merge DCR 中，上游会指定保留记录（Winner）和删除记录（Loser），但实际审核时未必完全按照这个输入执行。
- **属性可能需要从 Loser 取值回填 Winner**：例如最终保留张医生 1，但某些属性需采用张医生 2 的值，因此保留记录的最终属性是多来源综合结果，而不是简单保留 Winner 原样。
- **界面需展示三方对比**：审核界面会同时列出 DCR 提议值、Winner 当前值、Loser 当前值，逐字段让审核人判断如何取值，甚至支持属性交换。
- **业务原因**：代表提交时有自己的判断，但第三方审核时可能发现原本应删除的记录其实是“强者”或更应保留的主记录，因此需要灵活保留某个特定 MDM ID，同时保证属性正确。
- **核心难点**：这里不是简单规则引擎，而是“记录保留规则 + 属性取值规则 + 人工例外判断”的组合，导致定开量大、界面复杂、测试复杂。

### 📣 模块九：申诉（Appeal）及状态变化处理
- **申诉场景的存在**：若某个 HCP DCR 被拒绝，或处于 Undetermined，代表不能简单重复提单，而是需要通过补充材料进行申诉。
- **状态竞争问题**：申诉过程中可能出现状态变化，例如发起申诉时还是 Undetermined，但提交后该 DCR 已被拒绝或已审批通过，于是系统要按不同状态做不同处理。
- **处理逻辑**：如果申诉时记录已经通过，则无需再处理；如果已拒绝，则应重新流转给第三方复验。这里体现出异步状态变化带来的分支逻辑。
- **意义**：这类场景不是主流程图能直接看出来的，却会显著增加实现与测试复杂度。

### 📚 模块十：文档、代码与知识传承现状
- **当前最大问题不是“要不要做”，而是“现状资料是否拿得到”**：会议多次聚焦在历史文档与代码能否获取，因为没有这些资料，仅靠人员回忆无法还原全部逻辑。
- **回忆不足以支撑估算**：参会者明确表示，时间太久，很多细节靠记忆已经无法恢复；没有文档就不可能准确判断复杂度。
- **文档存在但质量存疑**：历史上应留有一版交接或梳理文档，但预计比较乱、不完整，而且很多细节逻辑可能没有写透。
- **文档存放与访问限制**：资料在客户电脑/存储空间中，可能只能在客户环境查看，不能轻易外发。客户曾开放一部分信息，但理解上并不完整。
- **代码比文档更关键但更难获得**：如果文档写得差，最终还得看代码。尤其要做实施级复刻或准确估算时，代码参考几乎不可避免。只是当前获取代码难度较高。
- **运维现状**：当前运维已交由外部大项目团队统一负责，并非原实施团队直接维护，这也提高了获取知识与代码的沟通成本。

### 💰 模块十一：预算、客户诉求与投标判断
- **客户明确要求“现在就做 Planning”**：与客户沟通后得知，不管怎样都要推进，而且业务侧已经有计划，到年底必须做出规划。
- **License 成本已基本沉没**：EBX License 的钱已经拨出，不存在简单通过“推迟”来节省 License 的空间。
- **客户希望分 1.5 至 2 年实施**：原因不是技术，而是年度 IT Budget 有限，希望分摊预算，避免一年拿不出足够实施费。
- **整体预算预期下降**：客户认为未来不会再是历史上三四百万级的大项目，更倾向认为是 100 万到 200 万美元量级的事情。
- **客户对 AI 开发有期待**：客户提到希望“用 AI 搓出来”，团队对此明显持保留态度，认为 AI 可用于辅助，但不能替代对复杂主数据定开逻辑的理解与严谨开发。
- **核心担忧**：如果客户和团队都低估复杂度，后续 Planning 与实施都会出现认知偏差，最终导致周期、预算、风险失控。

### ⚖️ 模块十二：方案方向——重做还是保留 EBX 的 Plan B
- **方案并非只有重做一条路**：会上提出，如果复杂度过高、改造风险大，不应默认重做，应该认真评估保留 EBX、优化 License 与实施成本平衡的 Plan B。
- **判断依据是复杂度、成本、风险、现状耦合度**：需要综合主数据层复杂度、历史投入、实施风险、软件成本和接口/系统耦合程度来决定。
- **实施路线需结合现状 Roadmap**：若 HCO/HCP 或不同主题域能拆分，可能可分批上线；若与 CRM、上下游接口绑得很紧，则可能只能整锅切换。
- **主数据项目的现实风险**：Mdm 项目往往不是价值创造型，而是历史问题修复型；拖 1~2 年时，期间上下游接口、系统状态、组织架构、供应商都可能变化，导致规划基础不断漂移。
- **拖期风险极高**：与会者基于经验指出，真正痛苦的往往不是主系统内逻辑，而是长周期实施过程中外围接口和系统不断变化，导致项目持续被动重做。

### 🛣️ 模块十三：Roadmap 与分批实施思路
- **是否可分主题域上线**：会议探讨能否先切 HCO 或先切 HCP，或者按 BU、按系统边界分批上线。
- **核心看接口与耦合**：是否能分批，不取决于意愿，而取决于现状是否可切。如果接口绑定过深，分批路线可能不现实。
- **Planning 阶段应回答的问题**：需要明确该改造哪里、以什么方式做、周期多长、是一次切换还是分批路线，以及两种方案（重做/保留）在软件成本和实施成本上的平衡。
- **先从易到难的思路**：若切分可行，应优先把相对独立、相对容易的主题先切出去，以降低整体风险。

### 🗣️ 模块十四：投标前客户沟通策略
- **本周目标是“先 Say Hi，不深入实施细节”**：周四会议先做关系建立与初步沟通，不急于在当场把所有实施问题讲透。
- **下周前争取形成 Findings**：建议在正式讲标前，先与客户对齐一次认知，分享团队对当前复杂度与风险的发现，让客户理解项目不是“简单照搬”。
- **避免在讲标时才首次认知对齐**：如果在讲标时客户才第一次听到复杂度与 Plan B，容易形成“你们想复杂了”的对话，不利于建立专业判断。
- **需要用事实而非空对空说服客户**：因此文档、代码、历史 CR、历史费用变化等都应成为支撑材料，帮助客户看到复杂度的客观依据。

## ✅达成的共识
- 当前项目的复杂度不能仅凭口头回忆判断，必须尽快获取历史文档，最好还能结合代码分析。
- 从现有回顾看，Master Layer/主数据定制逻辑在历史项目中占比很高，会议中有人判断大约可占 70%。
- 历史复杂度的核心并不只是流程数量，而是大量子场景、状态分支、Merge 联动、申诉、批量处理、界面定制及平台能力不足带来的定开。
- 客户今年必须推进 Planning，且倾向按 1.5 至 2 年实施分摊预算。
- 在投标或方案沟通中，不能只讲抽象方法论，必须把复杂度与风险“摊开来”讲。
- 后续需要同时保留两类方案思路：一类是基于现状重做的 Planning 路线；另一类是保留 EBX 或降低改造幅度的 Plan B。
- 周四先进行初步沟通，后续再安排一次基于 findings 的认知对齐。

## ❓未达成共识 / 仍需进一步确认
- 历史项目的完整文档是否能取得、能取得到什么粒度，目前仍未确认。
- 历史代码是否有机会查看或参考，获取难度较高，尚无明确路径。
- 仅凭文档是否足以完成复杂度估算，还是必须看代码，仍取决于文档质量。
- 客户对“复杂度到底有多高”的认知目前可能仍偏低，需要进一步验证和校正。
- 最终建议客户走重做方案还是保留 EBX 的替代方案，目前尚不能下结论。
- Roadmap 是否可以拆批、按 HCO/HCP 或 BU 分阶段上线，取决于接口与现状耦合度，仍待调研。

- [ ] 基于文档/代码梳理历史 DCR、Merge、Batch Review、Appeal、Match、关联关系等核心场景。
- [ ] 将“主流程 + 子场景 + 例外分支 + 界面复杂度”整理为对客户可展示的复杂度证据。
- [ ] 结合历史费用、历史 CR 变化、实施周期，建立对复杂度的正反向论证。
- [ ] 评估两类备选路线：重做方案与保留 EBX/降低改造幅度方案。
- [ ] 在正式讲标前，与客户做一次认知拉齐，分享团队 findings。
- [ ] 周四先完成初步客户接触与关系建立，不急于展开过深实施细节。
- [ ] 后续在 Planning 中补充对 Roadmap、分批实施、接口耦合与切换方式的分析。

## 💬引用观点
- "复杂度不是只有十几个场景，而是每个场景下面还有很多子场景和细节分支。"
- "中国区的设计永远是要人来处理，不会自动接受或者自动拒绝。"
- "客户其实想把 MDM 当成数据库，但又不得不保留一些数据质量控制。"
- "如果没有文档，靠回忆肯定回忆不全；如果文档不好，最终还是要看代码。"
- "我们不能空对空地跟客户讲复杂，得把复杂度摊开来给他看。"
- "如果客户和我们都低估了复杂度，后面做 Planning 和实施都会很危险。"
- "AI 可以做辅助，但不能指望它把这种复杂主数据逻辑直接搓出来。"
- "主数据项目很多时候不是创造价值，而是在修历史问题。"

## **🤖AI建议**
- **先证据化复杂度，再讨论方案优劣**：当前最需要的不是继续凭经验争论“到底复杂不复杂”，而是把历史文档、代码、CR、费用变化、场景清单转化为证据链。只有证据化后，客户才更容易接受较长周期、较高成本或保留 EBX 的建议。
- **把复杂度拆成四层来讲**：建议后续对外汇报时按“业务场景复杂度、数据模型与关联复杂度、平台定开复杂度、实施/切换风险复杂度”四层展开，这样比单纯说‘有很多场景’更有说服力。
- **对客户先做认知校准，再做方案推演**：在正式讲标前，先做一次 findings 分享，让客户理解历史实现为何贵、为何慢、为何风险高，再进入方案选择，能降低后续争议。
- **Plan A / Plan B 要并行准备**：一方面准备基于现状重做的 Planning 逻辑，另一方面准备保留 EBX 或有限改造的替代路线。这样既能显示团队务实，也能让客户看到成本—风险平衡思路。
- **尽量把 Roadmap 与切分条件说清楚**：不要直接承诺可分批或不可分批，而应明确：是否能按 HCO/HCP、BU、接口域切分，完全取决于现状接口和耦合分析。这会让 Planning 更专业，也避免后续承诺失真。
- **重视文档质量评估本身**：拿到文档后，不仅要看内容，还要给出‘文档是否足以支持估算/实施’的判断。如果文档不足，应明确提出必须看代码，避免用低质量输入做高风险估算。
- **谨慎使用 AI 的定位**：对内可考虑用 AI 辅助梳理文档、提取流程、总结代码结构，但对外不应承诺 AI 会显著压缩核心复杂度实现成本，以免形成不切实际预期。
- **Planning 交付物应覆盖决策问题**：最终 Planning 不只是出一份排期，而应回答：现状是什么、复杂度多高、可选路线有哪些、各路线的软件成本/实施成本/风险如何、推荐哪条路径以及为什么。
