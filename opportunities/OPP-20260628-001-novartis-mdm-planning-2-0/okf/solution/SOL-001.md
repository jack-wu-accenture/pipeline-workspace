## 背景

项目名"2.0"指第二轮招标。第一轮 Reltio 方案因 Global 成本过高被叫停，本次重新规划。
本次 RFP 范围 = **规划阶段**，以 Go/No-go 决策（Nov-Dec 2026）收尾，不含实施。

## 核心方案思路：最小替换，最大复用

**替换范围（仅 MDM 核心引擎）：**
- 实体管理（HCP/HCO CRUD、数据模型、版本控制）
- Match & Merge 核心（评分规则配置 + 合并决策）
- EBX API 层 → 新 REST API

**保持不动（外挂层）：**
- SCDF batch pipeline + Elasticsearch（批量去重，含 Matching 逻辑）
- 所有 Spring Boot/Cloud 外挂业务逻辑
- MFT/MuleSoft/Solace 集成层

集成路径：`Source → MFT/MuleSoft/Solace → SCDF → 新 MDM 核心 API`

## 技术栈

| 层 | 选型 |
|----|------|
| 实体存储 | RDS PostgreSQL（配置驱动数据模型，参考 Pimcore 模式）|
| 批量 Match | OpenSearch（AWS China 托管），复用 SCDF 逻辑 |
| 非批量 Matching | 自建评分服务，规则来自 NVS MDM.excalidraw |
| 外挂层 | 不变，仅替换 EBX API 调用为新引擎 REST API |
| 运行环境 | AWS China（宁夏/北京 region）|

## 竞争格局与差异化

**竞对**：Cognizant

**核心差异化**：Accenture 是 Novartis Global DCAM 服务供应商

WS-B 不只是出一份治理报告——战略目标是通过 DCAM 评审机制将 Global EDM 团队拉入方案评估过程，使其成为共同作者，在 Go/No-go 前已为 China 方案背书。Cognizant 无法复制这一优势。

**WS-B 执行方式（草案）：**
1. 与 Novartis Global EDM 团队对齐 MDM 方案的 DCAM capability 要求
2. 设计 DCAM 评估框架（针对 AWS 自建方案），邀请 Global 人员参与评分
3. 评估结果纳入 WS-C 方案对比报告，作为 Global 背书的正式记录
4. Go/No-go 会议上 Global EDM 作为共同推荐方出现

## WS-B 策略：Global EDM 从反对方 → 参与方

Global 现状：已知晓项目，不支持（合规风险），但无更好替代方案（MDM 产品成本问题无解）。
Accenture 角色：帮助 Global 找到参与路径 = 通过 DCAM 评审机制提升项目合规性。

执行方式（长周期连续会议）：
1. 解读 DCAM 框架，逐条映射为 China MDM 方案具体需求
2. Global EDM 人员参与每轮 review，验证映射结果
3. 产出：满足 DCAM 要求的 MDM 需求说明书（经 Global review 的正式文档）
4. Go/No-go 时 Global 已在局内，方案自然获得背书

## WS-E 核心验证：Child Data Space 去除

当前 EBX 使用 Child Data Space（子数据空间）管理 DCR 草稿版本，导致外挂层大量冗余设计。
新方案不引入此概念，但不能做简单接口替换——需自开发状态机替代。

PoC 选取典型场景（HCP DCR 创建/审批）：
- 验证去掉 Child Data Space 能否简化设计
- 验证替代方案（自开发状态机）开发成本
- 输出工作量估算，作为实施阶段报价依据
